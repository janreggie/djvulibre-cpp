\begin{Indent}\textbf{ J\+B2\+Image.\+h}\par
{\em Files \#\char`\"{}\+J\+B2\+Image.\+h\char`\"{}\# and \#\char`\"{}\+J\+B2\+Image.\+cpp\char`\"{}\# address the compression of bilevel images using the J\+B2 soft pattern matching scheme. These files provide the complete decoder and the decoder back-\/end. The J\+B2 scheme is optimized for images containing a large number of self-\/similar small components such as characters. Typical text images can be compressed into files 3 to 5 times smaller than with G4/\+M\+MR and 2 to 4 times smaller than with J\+B\+I\+G1.

\{\textbackslash{}bf J\+B2 and J\+B\+I\+G2\} --- J\+B2 has strong similarities with the forthcoming J\+B\+I\+G2 standard developed by the \char`\"{}\+I\+S\+O/\+I\+E\+C J\+T\+C1 S\+C29 Working Group 1\char`\"{} which is responsible for both the J\+P\+EG and J\+B\+IG standards. This is hardly surprising since J\+B2 was our own proposal for the J\+B\+I\+G2 standard and remained the only proposal for years. The full J\+B\+I\+G2 standard however is significantly more complex and slighlty less efficient than J\+B2 because it addresses a broader range of applications. Full J\+B\+I\+G2 compliance may be implemented in the future.

\{\textbackslash{}bf J\+B2 Images\} --- \mbox{\hyperlink{structClass}{Class}} \textbackslash{}\+Ref\{\mbox{\hyperlink{classJB2Image}{J\+B2\+Image}}\} is the central data structure implemented here. A \mbox{\hyperlink{classJB2Image}{J\+B2\+Image}}\# is composed of an array of shapes and an array of blits. Each shape contains a small bitmap representing an elementary blob of ink, such as a character or a segment of line art. Each blit instructs the decoder to render a particular shape at a specified position in the image. Some compression is already achieved because several blits can refer to the same shape. A shape can also contain a pointer to a parent shape. Additional compression is achieved when both shapes are similar because each shape is encoded using the parent shape as a model. A \#\char`\"{}\+O\char`\"{}\# shape for instance could be a parent for both a \#\char`\"{}\+C\char`\"{}\# shape and a \#\char`\"{}\+Q\char`\"{}\# shape.

\{\textbackslash{}bf J\+B2 Dictionary\} --- \mbox{\hyperlink{structClass}{Class}} \textbackslash{}\+Ref\{\mbox{\hyperlink{classJB2Dict}{J\+B2\+Dict}}\} is a peculiar kind of \mbox{\hyperlink{classJB2Image}{J\+B2\+Image}} which only contains an array of shapes. These shapes can be referenced from another J\+B2\+Dict/\+J\+B2\+Image. This is arranged by setting the \`{}`inherited dictionary'\textquotesingle{} of a J\+B2\+Dict/\+J\+B2\+Image using function \textbackslash{}\+Ref\{\mbox{\hyperlink{classJB2Dict_a775b7bf757ef6d27a08a48d142f66c22}{J\+B2\+Dict\+::set\+\_\+inherited\+\_\+dict}}\}. Several J\+B2\+Images can use shapes from a same \mbox{\hyperlink{classJB2Dict}{J\+B2\+Dict}} encoded separately. This is how several pages of a same document can share information.

\{\textbackslash{}bf Decoding J\+B2 data\} --- The first step for decoding J\+B2 data consists of creating an empty \mbox{\hyperlink{classJB2Image}{J\+B2\+Image}}\# object. Function \textbackslash{}\+Ref\{\mbox{\hyperlink{classJB2Image_ac108ac2e1f06336fc5ddaeeb36671a31}{J\+B2\+Image\+::decode}}\} then reads the data and populates the \mbox{\hyperlink{classJB2Image}{J\+B2\+Image}}\# with the shapes and the blits. Function \textbackslash{}\+Ref\{\mbox{\hyperlink{classJB2Image_ab3b2661c41345ebd2eab536485f32763}{J\+B2\+Image\+::get\+\_\+bitmap}}\} finally produces an anti-\/aliased image.

\{\textbackslash{}bf Encoding J\+B2 data\} --- The first step for decoding J\+B2 data also consists of creating an empty \mbox{\hyperlink{classJB2Image}{J\+B2\+Image}}\# object. You must then use functions \textbackslash{}\+Ref\{\mbox{\hyperlink{classJB2Dict_a9b5fd10e58af3d0f7b6dc7f133abd719}{J\+B2\+Image\+::add\+\_\+shape}}\} and \textbackslash{}\+Ref\{\mbox{\hyperlink{classJB2Image_ad742868b21e3ffc2c912e00a2e9b67d4}{J\+B2\+Image\+::add\+\_\+blit}}\} to populate the \mbox{\hyperlink{classJB2Image}{J\+B2\+Image}}\# object. Function \textbackslash{}\+Ref\{\mbox{\hyperlink{classJB2Image_af2b2723be7c37a94d3ad67cd72a9bfc9}{J\+B2\+Image\+::encode}}\} finally produces the J\+B2 data. Function \mbox{\hyperlink{classJB2Image_af2b2723be7c37a94d3ad67cd72a9bfc9}{encode}}\# sequentially encodes the blits and the necessary shapes. The compression ratio depends on several factors\+: \textbackslash{}begin\{itemize\} \textbackslash{}item Blits should reuse shapes as often as possible. \textbackslash{}item Blits should be sorted in reading order because this facilitates the prediction of the blit coordinates. \textbackslash{}item Shapes should be sorted according to the order of first appearance in the sequence of blits because this facilitates the prediction of the shape indices. \textbackslash{}item Shapes should be compared to all previous shapes in the shape array. The shape parent pointer should be set to a suitable parent shape if such a parent shape exists. The parent shape should have almost the same size and the same pixels. \textbackslash{}end\{itemize\} All this is quite easy to achieve in the case of an electronically produced document such as a D\+VI file or a PS file\+: we know what the characters are and where they are located. If you only have a scanned image however you must first locate the characters (connected component analysis) and cut the remaining pieces of ink into smaller blobs. Ordering the blits and matching the shapes is then an essentially heuristic process. Although the quality of the heuristics substantially effects the file size, misordering blits or mismatching shapes never effects the quality of the image. The last refinement consists in smoothing the shapes in order to reduce the noise and maximize the similarities between shapes.

\{\textbackslash{}bf J\+B2 extensions\} --- Two extensions of the J\+B2 encoding format have been introduced with Dj\+Vu files version 21. The first extension addresses the shared shape dictionaries. The second extension bounds the number of probability contexts used for coding numbers. Both extensions maintain backward compatibility with J\+B2 as described in the I\+C\+F\+DD proposal. A more complete discussion can be found in section \textbackslash{}\+Ref\{J\+B2 extensions for version 21.\}.

\doxysubsection{Detailed Description}
Main J\+B2 data structure. Each \mbox{\hyperlink{classJB2Image}{J\+B2\+Image}}\# consists of an array of shapes and an array of blits. These arrays can be populated by hand using functions \textbackslash{}\+Ref\{add\+\_\+shape\} and \textbackslash{}\+Ref\{add\+\_\+blit\}, or by decoding J\+B2 data using function \textbackslash{}\+Ref\{decode\}. You can then use function \textbackslash{}\+Ref\{get\+\_\+bitmap\} to render anti-\/aliased images, or use function \textbackslash{}\+Ref\{encode\} to generate J\+B2 data.
