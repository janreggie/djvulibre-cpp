\doxysubsection{Detailed Description}
\mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\# plays the central role in decoding \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuImage}{Dj\+Vu\+Image}}\}s. First of all, it represents a Dj\+Vu file whether it\textquotesingle{}s part of a multipage all-\/in-\/one-\/file Dj\+Vu document, or part of a multipage Dj\+Vu document where every page is in a separate file, or the whole single page document. \mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\# can read its contents from a file and store it back when necessary.

Second, \mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\# does the greatest part of decoding work. In the past this was the responsibility of \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuImage}{Dj\+Vu\+Image}}\}. Now, with the introduction of the multipage Dj\+Vu formats, the decoding routines have been extracted from the \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuImage}{Dj\+Vu\+Image}}\} and put into this separate class \mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\#.

As \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuImage}{Dj\+Vu\+Image}}\} before, \mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\# now contains public class variables corresponding to every component, that can ever be decoded from a Dj\+Vu file (such as \#\+I\+N\+FO\# chunk, \#\+B\+G44\# chunk, \#\+S\+J\+BZ\# chunk, etc.).

As before, the decoding is initiated by a single function (\textbackslash{}\+Ref\{start\+\_\+decode\}() in this case, and \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuImage_a24c91a159a1b4cae1f8b1d0d2055fe33}{Dj\+Vu\+Image\+::decode}}\}() before). The difference is that \mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\# now handles threads creation itself. When you call the \textbackslash{}\+Ref\{start\+\_\+decode\}() function, it creates the decoding thread, which starts decoding, and which can create additional threads\+: one per each file included into this one.

\{\textbackslash{}bf Inclusion\} is also a new feature specifically designed for a multipage document. Indeed, inside a given document there can be a lot of things shared between its pages. Examples can be the document annotation (\textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuAnno}{Dj\+Vu\+Anno}}\}) and other things like shared shapes and dictionary (to be implemented). To avoid putting these chunks into every page, we have invented new chunk called \#\+I\+N\+CL\# which purpose is to make the decoder open the specified file and decode it.

\{\textbackslash{}bf Source of data.\} The \mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\# can be initialized in two ways\+: \textbackslash{}begin\{itemize\} \textbackslash{}item With \#\+U\+RL\# and \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuPort}{Dj\+Vu\+Port}}\}. In this case \mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\# will request its data thru the communication mechanism provided by \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuPort}{Dj\+Vu\+Port}}\} in the constructor. If this file references (includes) any other file, data for them will also be requested in the same way. \textbackslash{}item With \textbackslash{}\+Ref\{\mbox{\hyperlink{classByteStream}{Byte\+Stream}}\}. In this case the \mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\# will read its data directly from the passed stream. This constructor has been added to simplify creation of \#\+Dj\+Vu\+File\#s, which do no include anything else. In this case the \textbackslash{}\+Ref\{\mbox{\hyperlink{classByteStream}{Byte\+Stream}}\} is enough for the \mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\# to initialize. \textbackslash{}end\{itemize\}

\{\textbackslash{}bf Progress information.\} \mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\# does not do decoding silently. Instead, it sends a whole set of notifications through the mechanism provided by \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuPort}{Dj\+Vu\+Port}}\} and \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuPortcaster}{Dj\+Vu\+Portcaster}}\}. It tells the user of the class about the progress of the decoding, about possible errors, chunk being decoded, etc. The data is requested using this mechanism too.

\{\textbackslash{}bf Creating.\} Depending on where you have data of the Dj\+Vu file, the \mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\# can be initialized in two ways\+: \textbackslash{}begin\{itemize\} \textbackslash{}item By providing \#\+U\+RL\# and pointer to \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuPort}{Dj\+Vu\+Port}}\}. In this case \mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\# will request data using communication mechanism provided by \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuPort}{Dj\+Vu\+Port}}\}. This is useful when the data is on the web or when this file includes other files. \textbackslash{}item By providing a \textbackslash{}\+Ref\{\mbox{\hyperlink{classByteStream}{Byte\+Stream}}\} with the data for the file. Use it only when the file doesn\textquotesingle{}t include other files. \textbackslash{}end\{itemize\} There is also a bunch of functions provided for composing the desired \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuDocument}{Dj\+Vu\+Document}}\} and modifying \mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\# structure. The examples are \textbackslash{}\+Ref\{delete\+\_\+chunks\}(), \textbackslash{}\+Ref\{insert\+\_\+chunk\}(), \textbackslash{}\+Ref\{include\+\_\+file\}() and \textbackslash{}\+Ref\{unlink\+\_\+file\}().

\{\textbackslash{}bf Caching.\} In the case of plugin it\textquotesingle{}s important to do the caching of decoded images or files. \mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\# appears to be the best candidate for caching, and that\textquotesingle{}s why it supports this procedure. Whenever a \mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\# is successfully decoded, it\textquotesingle{}s added to the cache by \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuDocument}{Dj\+Vu\+Document}}\}. Next time somebody needs it, it will be extracted from the cache directly by \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuDocument}{Dj\+Vu\+Document}}\} and won\textquotesingle{}t be decoded again.

\{\textbackslash{}bf U\+R\+Ls.\} Historically the biggest strain is put on making the decoder available for Netscape and IE plugins where the original files reside somewhere in the net. That is why \mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\# uses \{\textbackslash{}bf U\+R\+Ls\} to identify itself and other files. If you\textquotesingle{}re working with files on the hard disk, you have to use the local U\+R\+Ls instead of file names. A good way to do two way conversion is the \textbackslash{}\+Ref\{\mbox{\hyperlink{classGOS}{G\+OS}}\} class. Sometimes it happens that a given file does not reside anywhere but the memory. No problem in this case either. There is a special port \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuMemoryPort}{Dj\+Vu\+Memory\+Port}}\}, which can associate any U\+RL with the corresponding data in the memory. All you need to do is to invent your own U\+RL prefix for this case. \char`\"{}\#memory\+:\#\char`\"{} will do. The usage of absolute U\+R\+Ls has many advantages among which is the capability to cache files with their U\+RL being the cache key.

Please note, that the \mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\# class has been designed to work closely with \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuDocument}{Dj\+Vu\+Document}}\}. So please review the documentation on this class too.

Recursively stops all data-\/related operations.

Depending on the value of \#only\+\_\+blocked\# flag this works as follows\+: \textbackslash{}begin\{itemize\} \textbackslash{}item If \#only\+\_\+blocked\# is \#\+T\+R\+UE\#, the function will make sure, that any further access to the file\textquotesingle{}s data will result in a \#\+S\+T\+OP\# exception if the desired data is not available (and the thread would normally block). \textbackslash{}item If \#only\+\_\+blocked\# is \#\+F\+A\+L\+SE\#, then \{\textbackslash{}bf any\} further access to the file\textquotesingle{}s data will result in immediate \#\+S\+T\+OP\# exception. \textbackslash{}end\{itemize\}

The action of this function is recursive, meaning that any \mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\# included into this one will also be stopped.

Use this function when you don\textquotesingle{}t need the \mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\# anymore.
