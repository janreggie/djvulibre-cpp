\doxysubsection{Detailed Description}
\mbox{\hyperlink{classDjVuDocument}{Dj\+Vu\+Document}}\# provides convenient interface for opening, decoding and saving back Dj\+Vu documents in single page and multi page formats.

\{\textbackslash{}bf Input formats\} It can read multi page Dj\+Vu documents in either of the 4 formats\+: 2 obsolete (\{{\itshape old} bundled\} and \{{\itshape old} indexed\}) and two new (\{{\itshape new} bundled\} and \{{\itshape new} indirect\}).

\{\textbackslash{}bf Output formats\} To encourage users to switch to the new formats, the \mbox{\hyperlink{classDjVuDocument}{Dj\+Vu\+Document}}\# can save documents back only in the new formats\+: \{{\itshape bundled\}} and \{{\itshape indirect\}}.

\{\textbackslash{}bf Conversion.\} Since \mbox{\hyperlink{classDjVuDocument}{Dj\+Vu\+Document}}\# can open Dj\+Vu documents in an obsolete format and save it in any of the two new formats (\{{\itshape new} bundled\} and \{{\itshape new} indirect\}), this class can be used for conversion from obsolete formats to the new ones. Although it can also do conversion between the new two formats, it\textquotesingle{}s not the best way to do it. Please refer to \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVmDoc}{Dj\+Vm\+Doc}}\} for details.

\{\textbackslash{}bf Decoding.\} \mbox{\hyperlink{classDjVuDocument}{Dj\+Vu\+Document}}\# provides convenient interface for obtaining \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuImage}{Dj\+Vu\+Image}}\} corresponding to any page of the document. It uses \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuFileCache}{Dj\+Vu\+File\+Cache}}\} to do caching thus avoiding unnecessary multiple decoding of the same page. The real decoding though is accomplished by \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\}.

\{\textbackslash{}bf Messenging.\} Being derived from \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuPort}{Dj\+Vu\+Port}}\}, \mbox{\hyperlink{classDjVuDocument}{Dj\+Vu\+Document}}\# takes an active part in exchanging messages (requests and notifications) between different parties involved in decoding. It reports (relays) errors, progress information and even handles some requests for data (when these requests deal with local files).

Typical usage of \mbox{\hyperlink{classDjVuDocument}{Dj\+Vu\+Document}}\# class in a threadless command line program would be the following\+: \textbackslash{}begin\{verbatim\} static const char file\+\_\+name\mbox{[}\mbox{]}=\char`\"{}/tmp/document.\+djvu\char`\"{}; \mbox{\hyperlink{classGP}{G\+P$<$\+Dj\+Vu\+Document$>$}} doc=Dj\+Vu\+Document\+::create\+\_\+wait(file\+\_\+name); const int pages=doc-\/$>$\mbox{\hyperlink{classDjVuDocument_a935fe87ca93e07352662f6a4a96dfaad}{get\+\_\+pages\+\_\+num()}}; for(int page=0;page$<$pages;page++) \{ \mbox{\hyperlink{classGP}{G\+P$<$\+Dj\+Vu\+Image$>$}} dimg=doc-\/$>$get\+\_\+page(page); Do something \}; \textbackslash{}end\{verbatim\}

\{\textbackslash{}bf \mbox{\hyperlink{classComments}{Comments}} for the code above\} \textbackslash{}begin\{enumerate\} \textbackslash{}item Since the document is assumed to be stored on the hard drive, we don\textquotesingle{}t have to cope with \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuPort}{Dj\+Vu\+Port}}\}s and can pass \#\+Z\+E\+RO\# pointer to the \textbackslash{}\+Ref\{init\}() function. \mbox{\hyperlink{classDjVuDocument}{Dj\+Vu\+Document}}\# can access local data itself. In the case of a plugin though, one would have to implement his own \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuPort}{Dj\+Vu\+Port}}\}, which would handle requests for data arising when the document is being decoded. \textbackslash{}item In a threaded program instead of calling the \textbackslash{}\+Ref\{init\}() function one can call \textbackslash{}\+Ref\{start\+\_\+init\}() and \textbackslash{}\+Ref\{stop\+\_\+init\}() to initiate and interrupt initialization carried out in another thread. This possibility of initializing the document in another thread has been added specially for the plugin because the initialization itself requires data, which is not immediately available in the plugin. Thus, to prevent the main thread from blocking, we perform initialization in a separate thread. To check if the class is completely and successfully initialized, use \textbackslash{}\+Ref\{is\+\_\+init\+\_\+ok\}(). To see if there was an error, use \textbackslash{}\+Ref\{is\+\_\+init\+\_\+failed\}(). To know when initialization is over (whether successfully or not), use \textbackslash{}\+Ref\{is\+\_\+init\+\_\+complete\}(). To wait for this to happen use \textbackslash{}\+Ref\{wait\+\_\+for\+\_\+complete\+\_\+init\}(). Once again, all these things are not required for single-\/threaded program.

Another difference between single-\/threaded and multi-\/threaded environments is that in a single-\/threaded program, the image is fully decoded before it\textquotesingle{}s returned. In a multi-\/threaded application decoding starts in a separate thread, and the pointer to the \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuImage}{Dj\+Vu\+Image}}\} being decoded is returned immediately. This has been done to enable progressive redisplay in the Dj\+Vu plugin. Use communication mechanism provided by \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuPort}{Dj\+Vu\+Port}}\} and \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuPortcaster}{Dj\+Vu\+Portcaster}}\} to learn about progress of decoding. Or try \#dimg-\/$>$wait\+\_\+for\+\_\+complete\+\_\+decode()\# to wait until the decoding ends. \textbackslash{}item See Also\+: \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuFile}{Dj\+Vu\+File}}\}, \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuImage}{Dj\+Vu\+Image}}\}, \textbackslash{}\+Ref\{\mbox{\hyperlink{classGOS}{G\+OS}}\}. \textbackslash{}end\{enumerate\}

\{\textbackslash{}bf Initialization\} As mentioned above, the \mbox{\hyperlink{classDjVuDocument}{Dj\+Vu\+Document}}\# can go through several stages of initialization. The functionality is gradually added while it passes one stage after another\+: \textbackslash{}begin\{enumerate\} \textbackslash{}item First of all, immediately after the object is created \textbackslash{}\+Ref\{init\}() or \textbackslash{}\+Ref\{start\+\_\+init\}() functions must be called. \{\textbackslash{}bf Nothing\} will work until this is done. \textbackslash{}\+Ref\{init\}() function will not return until the initialization is complete. You need to make sure, that enough data is available. \{\textbackslash{}bf Do not call \textbackslash{}\+Ref\{init\}() in the plugin\}. \textbackslash{}\+Ref\{start\+\_\+init\}() will start initialization in another thread. Use \textbackslash{}\+Ref\{stop\+\_\+init\}() to interrupt it. Use \textbackslash{}\+Ref\{is\+\_\+init\+\_\+complete\}() to check the initialization progress. Use \textbackslash{}\+Ref\{wait\+\_\+for\+\_\+complete\+\_\+init\}() to wait for init to finish. \textbackslash{}item The first thing the initializing code learns about the document is its type (\#\+B\+U\+N\+D\+L\+ED\#, \#\+I\+N\+D\+I\+R\+E\+CT\#, \#\+O\+L\+D\+\_\+\+B\+U\+N\+D\+L\+ED\# or \#\+O\+L\+D\+\_\+\+I\+N\+D\+E\+X\+ED\#). As soon as it happens, document flags are changed and \mbox{\hyperlink{classDjVuPort_aa522666df6f9c65ffc8d161602f0fc2b}{notify\+\_\+doc\+\_\+flags\+\_\+changed()}}\# request is sent through the communication mechanism provided by \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuPortcaster}{Dj\+Vu\+Portcaster}}\}. \textbackslash{}item After the document type becomes known, the initializing code proceeds with learning the document structure. Gradually the flags are updated with values\+: \textbackslash{}begin\{itemize\} \textbackslash{}item \#\+D\+O\+C\+\_\+\+D\+I\+R\+\_\+\+K\+N\+O\+WN\#\+: Contents of the document became known. This is meaningful for \#\+B\+U\+N\+D\+L\+ED\#, \#\+O\+L\+D\+\_\+\+B\+U\+N\+D\+L\+ED\# and \#\+I\+N\+D\+I\+R\+E\+CT\# documents only. \textbackslash{}item \#\+D\+O\+C\+\_\+\+N\+D\+I\+R\+\_\+\+K\+N\+O\+WN\#\+: Contents of the document navigation directory became known. This is meaningful for old-\/style documents (\#\+O\+L\+D\+\_\+\+B\+U\+N\+D\+L\+ED\# and \#\+O\+L\+D\+\_\+\+I\+N\+D\+E\+X\+ED\#) only \textbackslash{}item \#\+D\+O\+C\+\_\+\+I\+N\+I\+T\+\_\+\+OK\# or \#\+D\+O\+C\+\_\+\+I\+N\+I\+T\+\_\+\+F\+A\+I\+L\+ED\#\+: The initializating code finished. \textbackslash{}end\{itemize\} \textbackslash{}end\{enumerate\}

Initializes the \mbox{\hyperlink{classDjVuDocument}{Dj\+Vu\+Document}}\# object using an existing document. This function should be called once after creating the object. The \#url\# should point to the real data, and the creator of the document should be ready to return this data to the document if it\textquotesingle{}s not stored locally (in which case \mbox{\hyperlink{classDjVuDocument}{Dj\+Vu\+Document}}\# can access it itself).

\{\textbackslash{}bf Initializing thread\} In a single-\/threaded application, the \mbox{\hyperlink{classDjVuDocument_af0725484fdb470903de22e463dddc559}{start\+\_\+init()}}\# function performs the complete initialization of the \mbox{\hyperlink{classDjVuDocument}{Dj\+Vu\+Document}}\# before it returns. In a multi-\/threaded application, though, it initializes some internal variables, requests data for the document and starts a new \{{\itshape initializing\}} thread, which is responsible for determining the document type and structure and completing the initialization process. This additional complication is justified in the case of the Dj\+Vu plugin because performing initialization requires data and in the plugin the data can be supplied by the main thread only. Thus, if the initialization was completed by the main thread, the plugin would run out of data and block.

\{\textbackslash{}bf Stages of initialization\} Immediately after the \mbox{\hyperlink{classDjVuDocument_af0725484fdb470903de22e463dddc559}{start\+\_\+init()}}\# function terminates, the \mbox{\hyperlink{classDjVuDocument}{Dj\+Vu\+Document}}\# object is ready for use. Its functionality will not be complete (until the initializing thread finishes), but the object is still very useful. Such functions as \textbackslash{}\+Ref\{get\+\_\+page\}() or \textbackslash{}\+Ref\{get\+\_\+djvu\+\_\+file\}() or \textbackslash{}\+Ref\{id\+\_\+to\+\_\+url\}() may be called before the initializing thread completes. This allows the Dj\+Vu plugin start decoding as soon as possible without waiting for all data to arrive.

To query the current stage of initialization you can use \textbackslash{}\+Ref\{get\+\_\+doc\+\_\+flags\}() function or listen to the \mbox{\hyperlink{classDjVuPort_aa522666df6f9c65ffc8d161602f0fc2b}{notify\+\_\+doc\+\_\+flags\+\_\+changed()}}\# notifications distributed with the help of \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuPortcaster}{Dj\+Vu\+Portcaster}}\}. To wait for the initialization to complete use \textbackslash{}\+Ref\{wait\+\_\+for\+\_\+complete\+\_\+init\}(). To stop initialization call \textbackslash{}\+Ref\{stop\+\_\+init\}().

\{\textbackslash{}bf Querying data\} The query for data is done using the communication mechanism provided by \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuPort}{Dj\+Vu\+Port}}\} and \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuPortcaster}{Dj\+Vu\+Portcaster}}\}. If \#port\# is not \#\+Z\+E\+RO\#, then the request for data will be forwarded to it. If it \{\textbackslash{}bf is\} \#\+Z\+E\+RO\# then \mbox{\hyperlink{classDjVuDocument}{Dj\+Vu\+Document}}\# will create an internal instance of \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuSimplePort}{Dj\+Vu\+Simple\+Port}}\} and will use it to access local files and report errors to \#stderr\#. In short, if the document file is stored on the local hard disk, and you\textquotesingle{}re OK about reporting errors to \#stderr\#, you may pass \#\+Z\+E\+RO\# pointer to \textbackslash{}\+Ref\{\mbox{\hyperlink{classDjVuPort}{Dj\+Vu\+Port}}\} as \mbox{\hyperlink{classDjVuDocument}{Dj\+Vu\+Document}}\# can take care of this situation by itself.

\{\textbackslash{}bf The U\+RL\} Depending on the document type the \#url\# should point to\+: \textbackslash{}begin\{itemize\} \textbackslash{}item \{\textbackslash{}bf Old bundled\} and \{\textbackslash{}bf New bundled\} formats\+: to the document itself. \textbackslash{}item \{\textbackslash{}bf Old indexed\} format\+: to any page of the document. \textbackslash{}item \{\textbackslash{}bf New indirect\} format\+: to the top-\/level file of the document. If (like in the \{{\itshape old} indexed\} format) you point the \#url\# to a page, the page \{{\itshape will\}} be decoded, but it will \{{\itshape not\}} be recognized to be part of the document. \textbackslash{}end\{itemize\}
